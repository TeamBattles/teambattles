## Overview

Webhooks allow your application to receive real-time HTTP POST notifications when events occur on TeamBattles. Instead of polling the API, you register an endpoint URL and select which events to receive.

## Setup

### Register an Endpoint

Go to [Settings > Developer](https://teambattles.gg/settings/developer) and click "Add Endpoint" in the Webhooks section, or use the API:

```bash
POST /api/user/webhooks
```

**Body:**

```json
{
	"url": "https://your-server.com/webhooks/teambattles",
	"events": ["match.started", "match.completed", "score.submitted"],
	"label": "Production Server"
}
```

### Webhook Secret

When you create an endpoint, a signing secret is generated. This secret is used to verify that webhook deliveries are from TeamBattles.

<Warning>
	The webhook secret is shown only once when the endpoint is created. Store it securely.
</Warning>

## Events

### Match Events

| Event             | Description                                      |
| ----------------- | ------------------------------------------------ |
| `match.created`   | A new match was created                          |
| `match.published` | Match was published and visible to opponents     |
| `match.accepted`  | An opponent accepted the match                   |
| `match.ready`     | Match is ready to begin (all check-ins complete) |
| `match.started`   | Match began                                      |
| `match.completed` | Match finished with a result                     |
| `match.cancelled` | Match was cancelled                              |
| `match.forfeited` | A team forfeited the match                       |

### Score Events

| Event             | Description                           |
| ----------------- | ------------------------------------- |
| `score.submitted` | A score was submitted                 |
| `score.confirmed` | A score was confirmed by the opponent |
| `score.disputed`  | A submitted score was disputed        |

### Team & Player Events

| Event               | Description                       |
| ------------------- | --------------------------------- |
| `roster.updated`    | Match roster was changed          |
| `player.checked_in` | A player checked in for the match |
| `team.joined`       | A team joined the match           |
| `team.left`         | A team left the match             |

## Payload Format

All webhook deliveries use the following format:

```json
{
	"id": "evt_abc123",
	"event": "match.completed",
	"created_at": "2026-02-23T15:30:00Z",
	"data": {
		"matchId": "match_123",
		"gameId": "call_of_duty_black_ops_7",
		"winner": "team_abc",
		"scores": {
			"team_abc": 250,
			"team_def": 180
		}
	}
}
```

## Signature Verification

Every delivery is signed using the [Standard Webhooks](https://www.standardwebhooks.com/) scheme via three headers:

| Header              | Description                                                                                                                                                       |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `webhook-id`        | The stable delivery id (also the envelope `id`), reused across retries. Use it to process events idempotently.                                                   |
| `webhook-timestamp` | Unix timestamp (seconds) of this attempt. Reject deliveries outside a tolerance window (e.g. 5 minutes) for replay protection.                                    |
| `webhook-signature` | One or more space-separated `v1,<base64>` signatures. During a signing-secret rotation grace window both the new and previous secret sign, so accept ANY of them. |

The signed content is `${webhook-id}.${webhook-timestamp}.${rawBody}`, HMAC-SHA256 with your endpoint's signing secret, base64-encoded. The `X-TeamBattles-Event` header carries the event name for convenience.

### Verifying in Node.js

```javascript
import crypto from "crypto";

function verifyWebhookSignature(headers, rawBody, secret) {
	const id = headers["webhook-id"];
	const timestamp = headers["webhook-timestamp"];
	const sigHeader = headers["webhook-signature"]; // "v1,<base64> [v1,<base64>]"

	// Reject stale deliveries (replay protection).
	if (Math.abs(Math.floor(Date.now() / 1000) - Number(timestamp)) > 300) return false;

	const signedContent = `${id}.${timestamp}.${rawBody}`;
	const expected = crypto.createHmac("sha256", secret).update(signedContent, "utf8").digest("base64");

	// Accept if ANY provided v1 signature matches (covers the rotation grace window).
	return sigHeader.split(" ").some((token) => {
		const [version, sig] = token.split(",");
		return (
			version === "v1" &&
			sig.length === expected.length &&
			crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(expected))
		);
	});
}

// In your webhook handler
app.post("/webhooks/teambattles", (req, res) => {
	if (!verifyWebhookSignature(req.headers, req.rawBody, process.env.WEBHOOK_SECRET)) {
		return res.status(401).send("Invalid signature");
	}
	const event = req.body;
	console.log(`Received ${event.event}:`, event.data);
	res.status(200).send("OK");
});
```

### Verifying in Python

```python
import hmac, hashlib, base64, time

def verify_signature(headers, raw_body: bytes, secret: str) -> bool:
    webhook_id = headers["webhook-id"]
    timestamp = headers["webhook-timestamp"]
    sig_header = headers["webhook-signature"]  # "v1,<base64> [v1,<base64>]"

    if abs(int(time.time()) - int(timestamp)) > 300:
        return False

    signed = f"{webhook_id}.{timestamp}.".encode() + raw_body
    expected = base64.b64encode(hmac.new(secret.encode(), signed, hashlib.sha256).digest()).decode()
    return any(
        t.split(",", 1)[0] == "v1" and hmac.compare_digest(t.split(",", 1)[1], expected)
        for t in sig_header.split(" ")
    )
```

## Delivery

Deliveries use **at-least-once** semantics with automatic retries. Every attempt carries a stable `webhook-id` (the envelope `id`) reused across retries, so process events idempotently (dedupe on `webhook-id`).

- Webhooks are delivered within seconds of the event. Your endpoint must respond with a `2xx` status within a 10-second timeout.
- Requests are sent with `redirect: 'manual'`; any `3xx` response counts as a failed attempt.
- **Retries:** a transient failure (network error, timeout, `5xx`, or `429`) is retried with exponential backoff (10s, 20s, 40s, ... capped at 6h) for up to **6 attempts**. A non-retriable `4xx` (other than `429`) fails immediately.
- **Dead-letter:** once retries are exhausted (or on a permanent failure) the delivery is marked `dead_lettered`. You can replay a dead-lettered delivery from the API once your endpoint is healthy.
- After **50 consecutive dead-lettered deliveries**, the endpoint is automatically disabled and you receive an in-app notification.

<Note>
	A successful delivery resets the consecutive-failure counter. Re-enable a disabled endpoint from
	Settings &gt; Developer (or via the API) once your server is healthy again.
</Note>

## Testing

Use the "Test" button next to any webhook endpoint in the dashboard to send a test event. This sends a `test.ping` event to verify your endpoint is working.
