## API Keys

All API requests require authentication via an API key passed as a Bearer token in the `Authorization` header.

```bash
curl -H "Authorization: Bearer tb_your_api_key_here" \
  https://teambattles.gg/api/v1/matches
```

### Creating an API Key

1. Go to [Settings > Developer](https://teambattles.gg/settings/developer)
2. Click "Create API Key"
3. Enter a descriptive label (e.g., "Production Server")
4. Select the permissions your integration needs
5. Click "Save" - the full API key is shown only once

<Warning>
	Copy your API key immediately after creation. It cannot be retrieved later. If you lose it, you
	must create a new one.
</Warning>

### Permission Scopes

Each API key carries granular permission scopes, and each scope uses one of four level shapes: most use access levels (`none`, `read`, `read-write`); read-only scopes use `none`/`read`; write-only scopes use `none`/`write`; and connection scopes use `none`/`enabled`.

| Category        | Scope                                      | Values                 | Description                            |
| --------------- | ------------------------------------------ | ---------------------- | -------------------------------------- |
| **Matches**     | `user_matches`                             | none, read, read-write | Access your match history              |
| **Matches**     | `team_matches`                             | none, read, read-write | Access team match history              |
| **Matches**     | `org_matches`                              | none, read, read-write | Access organization match history      |
| **Matches**     | `public_matches`                           | none, read             | Discover published open matches        |
| **Connections** | `discord`, `twitch`, `github`, `battlenet` | none, enabled          | Access linked platform data            |
| **Game**        | `scores`                                   | none, write            | Submit scores via API                  |
| **Game**        | `lifecycle`                                | none, read, read-write | Query/manage match lifecycle           |
| **Game**        | `ranks`                                    | none, read, read-write | Read player game ranks                 |
| **Users**       | `profile`                                  | none, read             | Read user profile data                 |
| **Teams**       | `profile`, `roster`                        | none, read, read-write | Read or manage team profile and roster |
| **Orgs**        | `profile`                                  | none, read, read-write | Read or manage organization profile    |
| **Billing**     | `read`                                     | none, read             | Read your billing and subscription     |
| **Tickets**     | `read`                                     | none, read             | Read your support tickets              |
| **Tickets**     | `write`                                    | none, write            | Create or update support tickets       |
| **Strategies**  | `read`                                     | none, read             | Read your saved strategies             |
| **Chat**        | `read`                                     | none, read             | Read match chat                        |
| **Chat**        | `write`                                    | none, write            | Send match chat messages               |
| **Stream**      | `read`                                     | none, read             | Read overlay and streaming config      |
| **Webhooks**    | `manage`                                   | none, enabled          | Create and manage webhook endpoints    |
| **Uploads**     | `upload`                                   | none, read, read-write | Mint short-lived upload URLs for images (requires read-write) |

#### League capabilities

Keys bound to a league (a League developer account) also carry a `leagues` block with three access scopes - `league_public` (read), `league_admin` (read or read-write), and `settings_read` (read) - plus an optional `leagues.capabilities` allowlist. Each capability is **write-only**: it grants its action only when explicitly set to `write`. The 13 capability keys are:

`scores`, `seasons`, `teams`, `members`, `bans`, `penalties`, `cooldowns`, `rules`, `points`, `branding`, `games`, `display_rules`, `tickets`.

<Note>
	Game API permissions (scores, lifecycle, ranks) are only available to approved Game Developers,
	and league capabilities only to approved League Developers. Webhook management requires General
	Developer access. League capabilities are write-only - they are ignored unless set to `write`.
</Note>

### Key presets

The create-key dialog ships two one-click permission presets. A preset is only selectable
when every connection in its required-connections list is linked on your account.

| Preset           | Scopes set                                                                                | Required connections |
| ---------------- | ----------------------------------------------------------------------------------------- | -------------------- |
| `battles-replay` | `matches.user_matches: read`, `connections.twitch: enabled`                               | Twitch               |
| `league-manager` | `matches.team_matches: read`, `leagues.league_public: read`, `leagues.league_admin: read` | none                 |

### How scopes combine

A request is authorized by checking the calling key against the endpoint's scope
requirement. Every requirement is one of six kinds:

| Kind               | What it checks                                                                 |
| ------------------ | ------------------------------------------------------------------------------ |
| `access`           | A category permission at a level (`read` / `read-write` / `write`).            |
| `game`             | A game scope (`scores` / `lifecycle` / `ranks`) on a game-developer key.       |
| `connection`       | A connection toggle (`discord` / `twitch` / `github` / `battlenet`) is `enabled`. |
| `leagueCapability` | A league capability (e.g. `scores`, `seasons`) - satisfiable **only** by a league-operator key. |
| `webhook`          | The `webhooks.manage` scope.                                                    |
| `anyOf`            | A set of requirements where **any one** match authorizes the request.          |

Most endpoints need a single scope. Some accept `anyOf` - for example, reading a league is
granted by **either** the `league_public` read scope **or** the `league_admin` access scope,
so a public-read key and a league-operator key can both call it.

### Principal types

Every API key is one of three principal types, which determines what it can act on:

- **Personal** - acts as you, over your own data (your matches, connections, profile).
- **Game developer** - bound to a single game; submits verified scores and drives the
  match lifecycle for that game.
- **League operator** - bound to a single league; performs league-admin actions using the
  league capability scopes.

A league capability can only be satisfied by a league-operator key - a game-developer or
personal key can never exercise one, even if the scope value is set.

### Plans and feature gates

API access is sold on three developer plans: **API Free**, **API Pro**, and **API
Business**. Three capabilities are gated behind a paid (**API Pro** or higher) plan:

| Feature              | Requires  | What it unlocks                                  |
| -------------------- | --------- | ------------------------------------------------ |
| `writes`             | API Pro   | Any write/mutating endpoint.                     |
| `webhooks`           | API Pro   | Creating and managing webhook endpoints.         |
| `leagueOperatorApi`  | API Pro   | The League Operator API.                         |

If your plan is too low you receive `403 error_api_plan_required`; if the specific feature
is not on your plan you receive `403 error_api_feature_required`.

<Note>
	A **league-operator** key is gated on the **league owner's** plan, not the key
	creator's - one owner purchase covers all of that owner's leagues.
</Note>

### Key Prefix

API keys start with `tb_` followed by a random string. When viewing your keys in the dashboard, only the prefix is shown for security.

## Error Responses

Authentication and API errors return standard HTTP status codes with a consistent format:

```json
{
	"error": "error_code",
	"details": "Additional information (optional)"
}
```

The `error` field is a stable, machine-readable code. Branch on `error`, not on the human-readable `details` message (which is optional and frequently absent on auth and permission failures).

### Common Error Codes

| Code                              | HTTP Status | Description                                          |
| --------------------------------- | ----------- | --------------------------------------------------- |
| `error_api_key_required`          | 401         | No API key provided in Authorization header         |
| `error_api_key_invalid`           | 401         | API key is invalid or revoked                       |
| `error_api_key_expired`           | 401         | API key has passed its expiry date                  |
| `error_api_key_muted`             | 403         | API key is temporarily muted                        |
| `error_api_key_permission_denied` | 403         | API key lacks required permission                   |
| `error_api_plan_required`         | 403         | Endpoint requires a higher API subscription plan    |
| `error_api_feature_required`      | 403         | Endpoint requires an API feature not on your plan   |
| `error_game_scope_mismatch`       | 403         | API key is not approved for the requested game      |
| `error_league_scope_mismatch`     | 403         | API key is bound to a different league              |
| `error_not_match_participant`     | 403         | User is not allowed to read that match              |
| `error_api_rate_limited`          | 429         | Rate limit exceeded (see rate-limit headers below)  |
| `error_team_membership_required`  | 403         | User is not an active member of the team            |
| `error_org_membership_required`   | 403         | User is not an active member of the organization    |
| `error_team_not_found`            | 404         | Team ID or slug not found                           |
| `error_org_not_found`             | 404         | Organization ID or slug not found                   |
| `error_league_not_found`          | 404         | League ID or slug not found                         |
| `error_league_game_not_found`     | 404         | League game configuration was not found             |
| `error_league_season_not_found`   | 404         | League season ID was not found for that league      |
| `error_invalid_request`           | 400         | Request body or parameters are invalid              |
| `error_invalid_cursor`            | 400         | Pagination cursor is invalid or expired             |
| `error_invalid_limit`             | 400         | Pagination `limit` out of range: 1-50 on `POST /leagues`, 1-100 elsewhere |
| `error_invalid_date_format`       | 400         | Date parameter is not valid ISO 8601                |
| `error_internal`                  | 500         | Internal server error                               |

### Rate Limiting

API responses carry rate-limit headers on both allowed and rejected requests:

| Header                  | Description                                              |
| ----------------------- | ------------------------------------------------------- |
| `X-RateLimit-Limit`     | Maximum requests allowed in the current window          |
| `X-RateLimit-Remaining` | Requests remaining in the current window                |
| `X-RateLimit-Reset`     | Seconds until the window resets                         |
| `Retry-After`           | Seconds to wait before retrying (only on a 429 response) |

When the limit is exceeded, the API returns `429` with `error_api_rate_limited`. Wait `Retry-After` seconds before retrying.
